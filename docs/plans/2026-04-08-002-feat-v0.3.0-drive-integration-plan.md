# feat: v0.3.0 — Drive auth surfaces in TUI

## Overview

Make Google Drive authentication a **first-class, visible, gated** part of the TUI workflow. Drive auth currently happens silently inside the pipeline worker thread mid-export, with no indication to the user that it's happening, no visibility of auth state, and no error surface when it fails. This release fixes that.

This is the **first implementation cycle on the path to v1.0.0**, and it is intentionally narrow in scope: it does **not** redesign the Connect tab, does **not** add a Source picker, and does **not** introduce phone-local export. Those are downstream of v0.4.0 phone-local discovery and ship in v0.5.0.

## Roadmap Context

- **Roadmap doc**: [2026-04-08-001-feat-v1.0.0-public-release-roadmap.md](2026-04-08-001-feat-v1.0.0-public-release-roadmap.md)
- **Target release**: v0.3.0
- **Predecessor**: v0.2.0 (TUI discovery + export robustness)
- **Successor**: v0.4.0 (phone-local export discovery + prototype)
- **Why this release first**: smallest end-to-end vertical slice that improves user experience without depending on phone-local discovery findings.

## Problem Frame

Today, the TUI experience around Google Drive is broken in three ways:

1. **Invisible auth.** When the pipeline reaches Phase 1 (Download from Drive), `GoogleDriveManager.connect()` is called inside a worker thread. If a token exists, auth is silent and works. If no token exists, the OAuth `InstalledAppFlow.run_local_server()` opens a browser window in the background — the user doesn't see this happen, has to notice the browser, switch to it, authenticate, then switch back, all while the pipeline is already mid-run. If `client_secrets.json` is missing entirely, the pipeline errors out with a generic message and the user has no idea what to fix.

2. **No prerequisite gate.** The user can click "Start Export" without Drive being authenticated, without `client_secrets.json` being present, and without (in transcription mode) a valid OpenAI/ElevenLabs API key. Failures happen at the worst possible moment: after the user has waited for the export to finish.

3. **No status surface.** There is nowhere in the TUI that says "Drive: signed in as `<email>`" or "Drive: not signed in". The user has no way to verify their setup before committing to a workflow run.

A secondary, related problem: the `delete_from_drive` setting on the Settings panel is destructive — it permanently deletes WhatsApp exports from the user's Drive after each successful processing run — and is currently invisible to the user because Drive itself is invisible. The user can opt in to permanent deletion without ever realising Drive is involved.

## Requirements

- **R1.** The Settings panel grows a Drive section showing current Drive auth status and the path to `client_secrets.json` (or "not set"). The section includes a button to authenticate (or re-authenticate) and a file picker for the user's `client_secrets.json`.
- **R2.** When the user clicks "Sign in to Drive" from the Settings panel, the OAuth flow runs in a worker thread. The activity log narrates each step ("Opening browser…", "Waiting for authorization…", "Signed in as `<email>`"). On success the Settings panel updates to show the signed-in account.
- **R3.** When the user picks a `client_secrets.json` file via the file picker, it is copied into `~/.whatsapp_export/client_secrets.json` and the Settings panel updates to reflect the new path. The volume mount path remains the documented baseline for headless and Docker users; the file picker is convenience over the mount.
- **R4.** The Select tab grows a **Prerequisites** banner section. Each banner is a read-only status row that shows the state of one prerequisite. Banners explicitly tell the user where to change the relevant setting (e.g. "Drive: not signed in — open Settings tab to configure"). Banners only appear for prerequisites that the user's current settings actually require:
  - **Drive auth banner**: appears whenever the export path requires Drive (i.e. always, in v0.3.0 since Drive is the only path).
  - **OpenAI key banner**: appears whenever transcription is enabled and `whisper` is the chosen provider.
  - **ElevenLabs key banner**: appears whenever transcription is enabled and `elevenlabs` is the chosen provider.
  - **WhatsApp version banner**: appears whenever the connected WhatsApp version differs from the tested version (warn only, never blocks).
- **R5.** "Start Export" is a **hard gate**: the button is disabled whenever any required prerequisite is unmet. The currently-blocking prerequisites are listed inline so the user knows what to fix.
- **R6.** On device connection, the driver reads the installed WhatsApp version (`adb shell dumpsys package com.whatsapp | grep versionName`) and stores it on the app instance. The Select tab's WhatsApp version banner reflects whether the version matches the tested version. The tested version is held as a constant in code and updated by hand each release.
- **R7.** The `delete_from_drive` setting stays on the Settings panel as the source of truth. The Drive section on the Settings panel and the Drive banner on the Select tab both **show the current state** of `delete_from_drive` so the user understands the consequences. The first time in a session that the user enables `delete_from_drive`, a confirmation modal explains what will happen and requires explicit confirmation.
- **R8.** Drive auth errors (no `client_secrets.json`, OAuth flow cancelled, network failure, refresh token revoked) surface in the activity log with clear, actionable messages — what failed, why, what to try.
- **R9.** When the pipeline runs and Drive operations happen, the Summary pane's progress shows clearly which Drive operation is in flight ("Polling Drive for exports…", "Downloading X exports…", "Deleted Y exports from Drive"). No silent waits.
- **R10.** Existing headless mode behaviour is unchanged. Headless still uses the volume mount path for `client_secrets.json` and still does not have an interactive auth flow. Per-flag and per-arg behaviour stays the same.

## Key Technical Decisions

### D1. Drive auth lives in a worker thread, not the main event loop

`run_oauth_flow()` is blocking (it spins up a local HTTP server and waits for the user to authorize in their browser). It must run in a worker thread so the TUI stays responsive. The worker uses `thread=True` (not `asyncio`) to match the existing pattern in `connect_pane.py` and `discover_select_pane.py`. State changes go through `self.app.call_from_thread(...)` for UI updates.

### D2. Drive auth state lives on the app instance

Add `_drive_credentials: Optional[Credentials] = None` and `_drive_user_email: Optional[str] = None` to `WhatsAppExporterApp`. Provide read-only properties `drive_credentials` and `drive_user_email`. Banners and the Settings panel both read from these properties.

### D3. WhatsApp version detection happens on the driver, not in the TUI

`WhatsAppDriver` grows a `get_whatsapp_version()` method that reads `adb shell dumpsys package com.whatsapp | grep versionName`. The Connect pane calls this after `driver.connect()` succeeds and stores the result on the app instance as `_whatsapp_version`. The Select tab banner reads from there.

The **tested version** is a constant in code (e.g. `whatsapp_chat_autoexport/constants.py:TESTED_WHATSAPP_VERSION = "2.25.x.y"`) updated by hand each release. v0.7.0 README rewrite formalises this as a release checklist item.

### D4. Banners are read-only, Settings panel is the source of truth

Banners on the Select tab show status only — they do not contain input fields. When a banner needs the user to take action, it explicitly says "open Settings tab to configure". This is the agreed pattern from the roadmap and avoids having auth/key configuration UI in two places.

This is the v1.0.0 roadmap's locked decision: banners are status, Settings is configuration.

### D5. Hard gate on Start Export with inline reason

The Start Export button stays disabled whenever any required prerequisite is unmet. Below the button (or in the existing Selected count line) is a one-line summary of the current blocker(s): "Cannot export — Drive not signed in" / "Cannot export — OpenAI API key invalid" / "Cannot export — Drive not signed in, OpenAI key missing". The summary is generated from the list of active banners that are currently in an unmet state.

### D6. `delete_from_drive` first-time confirmation modal

The first time in a session that the `delete_from_drive` checkbox is checked, push a `CancelModal`-style modal explaining what will happen. The modal has two buttons: "I understand, delete after each export" (confirms) and "Cancel" (un-checks the box). The "I understand" choice is remembered for the rest of the session — subsequent toggles in the same session don't re-prompt. Across runs the checkbox state is not persisted, so the next launch starts unchecked.

### D7. WhatsApp version warning is non-blocking

The version check is **never** a hard gate. It's a banner that says "tested with X, you have Y, things may break". Start Export remains clickable. This is the locked roadmap decision.

### D8. Activity log narrates Drive operations during pipeline runs

Add `on_progress` callbacks (or extend the existing ones) on `drive_manager.connect()`, `wait_for_new_export()`, `batch_download_exports()`, and `delete_files()`, so the Summary pane's pipeline progress shows which Drive operation is in flight. Existing pipeline progress callback infrastructure already supports this — just add the new phase strings ("drive_connect", "drive_poll", "drive_download", "drive_delete") to the phase display map.

### D9. Settings persistence is OUT OF SCOPE for v0.3.0

Discussed in the roadmap as an open question. For v0.3.0 the user re-enters their `client_secrets.json` path on each launch (or relies on the volume mount which is naturally persistent). Adding settings persistence is a separate concern — touches every Settings field, not just Drive — and gets a dedicated decision in v0.4.0 or v0.5.0.

**Mitigation**: the file picker copies the user's selection to `~/.whatsapp_export/client_secrets.json`, which *is* persistent across runs. So while the Settings panel doesn't remember "the user said yes to Drive", the file itself is on disk and `auth.has_valid_token()` will pick up the cached token on next launch automatically. The UX cost is one extra click of "Sign in to Drive" per launch, which is acceptable for v0.3.0.

## Risks

- **Risk 1**: The OAuth `run_local_server()` flow opens a browser via the Python `webbrowser` module. In a Docker container (especially headless), this won't work. **Mitigation**: detect the no-browser case (TTY only, no DISPLAY) and instead surface a manual auth URL the user can open elsewhere. This is `flow.run_local_server(open_browser=False)` + reading `flow.authorization_url()`. **Caveat**: this is a non-trivial UX change that may not fit in v0.3.0. If it doesn't fit, document the limitation: Drive sign-in via the TUI requires a desktop environment; headless and Docker users use the volume-mount approach with a pre-existing token.

- **Risk 2**: The Textual file picker widget is `DirectoryTree` based and may not have a great UX for picking a single file. If the built-in picker isn't usable, fall back to a text input where the user types/pastes a path, with validation.

- **Risk 3**: WhatsApp version detection via `adb shell dumpsys package com.whatsapp` may have inconsistent output across Android versions. **Mitigation**: parse defensively (regex `versionName=([\d.]+)`), return `None` if parse fails. The banner correctly handles `None` ("WhatsApp version: unknown — proceed with caution").

- **Risk 4**: The hard gate logic gets complex when more prerequisite types are added in v0.5.0+. **Mitigation**: design `_compute_blocking_prerequisites()` as a list of small predicate functions from the start, so adding new prerequisites is appending to a list, not editing a monolithic conditional.

- **Risk 5**: The Drive section grows the Settings panel significantly, possibly making it scroll or feel cramped. **Mitigation**: prepare to refactor the Settings panel into collapsible sections in a follow-up if needed. v0.3.0 just adds the section; layout polish is acceptable as a fast-follow.

- **Risk 6**: Settings panel Drive section and Select tab banner can drift out of sync if state updates are missed. **Mitigation**: single source of truth on the app instance, single refresh entry point (`_refresh_banners` and `SettingsPanel._refresh_drive_section`), call both on the same triggers.

## Acceptance Criteria

A v0.3.0 release is acceptable when:

1. **Fresh install (no token, no client_secrets)**: launching the TUI, connecting a device, navigating to Select tab shows the Drive prerequisite banner in unmet state with text directing the user to the Settings tab. Start Export is disabled with the inline reason "Cannot export — Drive not signed in".
2. **Settings → choose file**: file picker works, copies to `~/.whatsapp_export/client_secrets.json`, Drive section updates to "ready to authenticate".
3. **Settings → sign in**: browser opens, user authenticates, returns to TUI, Drive section shows "Signed in as `<email>`".
4. **Back on Select tab**: Drive banner is now in ok state (or hidden), Start Export becomes enabled (assuming chats are selected).
5. **Cached token re-launch**: launching the TUI again skips the sign-in step; Drive section immediately shows "Signed in as `<email>`".
6. **Pipeline run**: activity log narrates each Drive operation by phase. No silent waits.
7. **Enable `delete_from_drive` for the first time**: confirmation modal appears, explains the consequence, requires explicit confirmation. Re-toggling in the same session does not re-prompt.
8. **WhatsApp version mismatch**: connecting a device whose WhatsApp version differs from `TESTED_WHATSAPP_VERSION` shows the warning banner; Start Export is **not** blocked.
9. **Connection without WhatsApp installed (or version unreadable)**: WhatsApp version banner shows "version unknown" and is non-blocking.
10. **Headless mode unchanged**: `whatsapp --headless --output ... --auto-select` works exactly as it did in v0.2.0, including with a pre-existing volume-mounted `client_secrets.json` and cached token.
11. **All existing tests still pass**. Unit test count goes up by ≥4 new test files; integration test count goes up by ≥1.
12. **Manual QA on real device** confirms each acceptance criterion above.
