# feat: v1.0.0 public release roadmap

## Purpose

This document is the **master roadmap** from the current `v0.2.0` baseline to a public `v1.0.0` release. It exists to:

1. Sequence the work across the `v0.3.x` → `v0.8.x` minor versions so that decisions made in early versions don't have to be undone later.
2. Make the dependencies between versions explicit (e.g. why phone-local export discovery has to happen before the Connect-tab redesign).
3. Hold a single source of truth for the v1.0.0 scope so per-version plans can cite it instead of re-litigating decisions.
4. Surface the **v1.0.0 risk register** in one place — particularly the things that could force a re-scope or timeline slip.

This is **not** an implementation plan. Each minor version has (or will have) its own per-version plan document. This roadmap describes the *shape* of those plans, not their contents.

## Scope of v1.0.0

A `v1.0.0` release is considered "publicly viable" when the following are true:

- **Anyone with the documented prerequisites can install it** (Docker primary, PyPI secondary).
- **Anyone with appropriate skills can contribute** (CONTRIBUTING.md, issue/PR templates, code of conduct, semantic-release wired up).
- **The tool is honest about what it does** (privacy section, WhatsApp ToS disclaimer, fragility note with tested WhatsApp version, runtime version warning).
- **The tool works without depending on Google Drive being on the critical path** (phone-local export is implemented as a first-class alternative to Drive transit).
- **Drive support is robust when used** (auth surfaced in TUI, errors visible, prerequisites gated before export starts).
- **Failures are visible and actionable** (prerequisite doctor on startup, error message audit, structured logging).

## Non-Goals for v1.0.0

These are deferred to v1.x or beyond. Calling them out so they don't sneak into per-version plans:

- **Public OAuth client / "click to sign in to Google" UX.** v1.0 requires BYO `client_secrets.json`. A public client is `v1.2.0` or later.
- **Cloud destinations other than Google Drive.** No Dropbox, S3, OneDrive, or rclone support in v1.0. Phone-local export is the only Drive-alternative shipped.
- **Drive folder picker.** Drive polling stays at the My Drive root for v1.0.
- **Multi-device support / parallel exports.** One device at a time, as today.
- **GUI / web UI.** Terminal TUI is the only UI surface.
- **Auto-update mechanism.** Users update by pulling a new Docker image or `pip install --upgrade`-ing.
- **Telemetry of any kind**, opt-in or otherwise.
- **TUI first-run banner for ToS / privacy notice.** README sections only for v1.0; in-TUI banner is a v1.x consideration.
- **Per-WhatsApp-version compatibility table maintained over time.** v1.0 documents *one* tested version. Multi-version compatibility tracking is a maintenance discipline that accrues post-v1.0.

## Distribution Decisions (locked)

| Decision | Choice |
|---|---|
| **Primary distribution** | Docker image, published to **GHCR** (GitHub Container Registry) under the project repo. |
| **Secondary distribution** | PyPI wheel + sdist, published as `whatsapp-chat-autoexport`. |
| **Standalone binaries** | Not shipped. |
| **macOS USB passthrough** | Not supported. Wireless ADB only. |
| **Docker on macOS workflow** | Wireless ADB. README documents this as the only macOS path. |
| **Docker on Linux workflow** | USB or wireless ADB. README documents both. |
| **Token / credentials directory** | `~/.whatsapp_export/` host directory, mounted into Docker as `-v ~/.whatsapp_export:/root/.whatsapp_export`. |

## Authentication & Credentials Decisions (locked)

| Decision | Choice |
|---|---|
| **Drive auth flow** | OAuth2.0 installed-app with browser popup (existing `auth.py` flow). |
| **Client credentials** | BYO `client_secrets.json` placed at `~/.whatsapp_export/client_secrets.json`. **Hard requirement**, no fallback. |
| **Public OAuth client** | Deferred to **v1.2.0**. Not a v1.0 blocker. |
| **TUI credential setup** | File picker on the Settings tab that copies the user's selected `client_secrets.json` into `~/.whatsapp_export/`. Volume mount remains the documented baseline for headless/Docker; the picker is convenience over the mount, not a replacement. |
| **Token storage** | Existing pickled `Credentials` at `~/.whatsapp_export/google_credentials.json`. No change. |
| **Scope** | `https://www.googleapis.com/auth/drive` (existing). No reduction in v1.0 — scope reduction is coupled to public OAuth client work and ships in v1.2.0+. |

## TUI UX Decisions (locked)

| Decision | Choice |
|---|---|
| **Drive auth gate** | Hard gate. Start Export is disabled until all prerequisites for the user's chosen settings are met. Refusing to start an unwinnable export is honest. |
| **Prerequisites surface** | Banners on the **Select tab** showing read-only status for whichever prerequisites the user's current settings require. Banners explicitly tell users where to change settings (e.g. "Open Settings tab to configure"). The Settings panel is the single source of truth for actual configuration. |
| **WhatsApp version check** | On connect, read installed WhatsApp version. Warn (never block) if the version differs from the tested version. Banner-style notice. |
| **`delete_from_drive` setting** | Source of truth on Settings panel (existing location). Drive section/banner shows current state. First-time confirmation modal when checked. |
| **Source picker** | Connect tab presents the user with a Source choice. v0.3.0 ships with **Drive** as the only option (no picker UI yet); v0.5.0 introduces the picker with **Drive**, **Phone-local**, and **Manual local-files** as choices. |

## Discovery / Research Required Before Implementation

These items require **discovery sessions** (poking at real hardware, capturing findings) before their per-version plans can be written:

- **Phone-local export feasibility** — see [docs/discovery/2026-04-08-001-phone-local-export-discovery.md](../discovery/2026-04-08-001-phone-local-export-discovery.md). This is the **single biggest risk to the v1.0 timeline**. If discovery surfaces blockers (no reliable share target across Android versions, scoped storage permission issues that require root, etc.), there is an explicit checkpoint at the end of v0.4.0 where we decide whether to proceed with v0.5.0 implementation or re-scope phone-local out of v1.0.

## Version Sequence

The sequence is ordered so that each version's decisions don't have to be undone in the next.

### v0.3.0 — Drive auth surfaces in TUI (no Connect-tab redesign)

**Scope**: Make Drive auth visible, gated, and configurable from inside the TUI. The Connect tab stays as it is today (device discovery only). Drive remains the only export path.

**Why first**: It's the smallest end-to-end vertical slice that improves user experience without depending on phone-local discovery.

**Why no Connect-tab redesign yet**: The Connect-tab redesign needs the Source picker, and the Source picker needs phone-local export to be a real choice, and phone-local export needs discovery first. Trying to redesign the Connect tab now would either require building a Source picker with only one option (wasted UI work) or building a Drive-centric design that gets thrown away in v0.5.0.

**Per-version plan**: [2026-04-08-002-feat-v0.3.0-drive-integration-plan.md](2026-04-08-002-feat-v0.3.0-drive-integration-plan.md)

### v0.4.0 — Phone-local export discovery + prototype

**Scope**: A focused research-and-prototyping cycle. Output is a working proof-of-concept that picks a non-Drive share target on the test device and ADB-pulls the resulting ZIP, plus a fully filled-in discovery document.

**Why second**: Phone-local export is a v1.0 blocker (decision locked). Its findings determine the architecture of the Connect-tab redesign in v0.5.0, so it has to happen before v0.5.0.

**Why a prototype rather than a plan**: We don't know enough yet to write a faithful implementation plan. The prototype's job is to surface the unknowns (which share targets exist, where files land, permission issues, edge cases) so a real plan can be written.

**Decision checkpoint at end of v0.4.0**: Based on prototype findings, decide one of:
- (a) **Proceed** to v0.5.0 implementation as planned. Phone-local export ships in v1.0.
- (b) **Re-scope v0.5.0** to handle a narrower set of devices/Android versions. Phone-local ships in v1.0 but with documented limitations.
- (c) **Defer phone-local out of v1.0** to a later v1.x. Drive remains the only export path in v1.0 and the Source picker shrinks to "Drive" + "Manual local-files". The v1.0 timeline shortens by one minor version.

This checkpoint is the **single most important decision point in the roadmap**.

**Discovery doc**: [docs/discovery/2026-04-08-001-phone-local-export-discovery.md](../discovery/2026-04-08-001-phone-local-export-discovery.md)

### v0.5.0 — Phone-local export production + Connect-tab Source picker

**Scope** (pending v0.4.0 outcome):

- Implement phone-local export based on v0.4.0 findings.
- Redesign the Connect tab to host the Source picker. Source choices: **Phone via ADB → Drive**, **Phone via ADB → Phone-local**, **Manual local-files folder**.
- Wire local-files mode (the existing `--pipeline-only` capability) into the picker as the third option. This is small but lives here because it shares the picker.
- Update the Drive prerequisite gate from v0.3.0 so it only triggers when the chosen Source actually requires Drive auth.
- Ensure WhatsApp version warning still fires for any Source choice that involves the device.

**Per-version plan**: To be written after v0.4.0 checkpoint.

### v0.6.0 — Hardening

**Scope**:
- **Prerequisite doctor**: on TUI startup (and in headless mode), check that `appium`, `adb`, and any other required CLI tools are on PATH. Surface clear, actionable errors at the earliest possible moment, not at the moment they're first invoked.
- **Error message audit**: walk every user-facing error path and rewrite messages to (a) say what failed, (b) say why, (c) say what to try. No raw stack traces in the activity log.
- **Version pinning**: pin Appium version, UiAutomator2 driver version, all Python dependencies. Document the pinned versions in the README.
- **Structured logging cleanup**: ensure all logger calls use consistent formatting and that the activity log surfaces useful state changes without being noisy.
- **WhatsApp version check** is from v0.3.0 — this is just a chance to revisit it after several versions of usage and tighten the UX if needed.

**May get pulled forward**: if any of these items are tightly coupled to v0.3.0 / v0.4.0 / v0.5.0 work, they get done in those versions instead. v0.6.0 is the cross-cutting catch-up pass.

### v0.7.0 — Documentation

**Scope**:
- **README rewrite**: lead with Docker quickstart, separate "I already have Appium" path for PyPI users, prerequisites table, install instructions.
- **Privacy section**: explicit statement of what data is processed, where it goes, what is sent to third parties (OpenAI/ElevenLabs for transcription, Google for Drive operations).
- **WhatsApp ToS disclaimer**: explicit notice that UI automation of WhatsApp may violate WhatsApp's terms of service; tool is not affiliated with WhatsApp; use at your own risk.
- **Fragility section**: explicit notice that UI selectors may break with WhatsApp updates; the **specific WhatsApp version** the release was tested against; how to file a bug report when selectors break.
- **Troubleshooting**: common failure modes and their fixes (Appium not on PATH, ADB device not authorized, Drive auth failing, transcription API key invalid, etc.).
- **Install paths**: Docker (primary, GHCR pull command), PyPI (secondary, pip command), poetry (development).
- **Configuration reference**: every flag, every setting, every environment variable.

### v0.8.0 — CI + release automation + contributor docs

**Scope**:
- **GitHub Actions**: lint + test on push/PR. Build Docker image. Build PyPI artifacts.
- **GHCR publishing**: on tag, build and push the Docker image to `ghcr.io/<owner>/whatsapp-chat-autoexport:<tag>` and `:latest`.
- **PyPI publishing**: on tag, build and push the wheel + sdist via PyPI Trusted Publishers (no API tokens stored).
- **Semantic-release / changelog automation**: pick a tool (probably `python-semantic-release` or `commitizen`), wire it up, generate `CHANGELOG.md` from commit history.
- **CONTRIBUTING.md**: how to set up dev env, how to run tests, how to write commit messages, how to open a PR.
- **Issue templates**: bug report (with required fields including WhatsApp version, Android version, Docker/PyPI install method), feature request, question.
- **PR template**: link to issue, what changed, how to test, screenshots if UI.
- **Code of conduct**: standard Contributor Covenant or similar.

### v1.0.0 — Public release

**Not feature work**. This version's job is:
- Final QA pass on a fresh machine (Docker on Linux + Docker on macOS via wireless + PyPI install on Linux).
- Bump version, update CHANGELOG.
- Tag, push, let the v0.8.0 release automation publish to GHCR + PyPI.
- Cut a GitHub Release with release notes.
- Write and publish announcement (where? README badge linking to release; optionally Reddit / Hacker News / lobste.rs; that's a separate question for closer to release time).

## Risk Register

| # | Risk | Severity | Mitigation |
|---|---|---|---|
| **R1** | **Phone-local export discovery surfaces blockers** that prevent reliable cross-device implementation. | **High** | v0.4.0 explicit checkpoint with three documented re-scope paths. v1.0 release is not gated on phone-local working perfectly — only on it being honestly documented. |
| **R2** | **WhatsApp UI changes** between writing v0.3.0 code and v1.0.0 release, breaking selectors. | Medium | Pin tested WhatsApp version. README sets expectations. v0.6.0 prerequisite doctor includes WhatsApp version check. Accept that this risk persists post-v1.0 and is a maintenance burden. |
| **R3** | **macOS Docker wireless ADB workflow** is not as reliable as we hope. | Medium | Test early — do this during v0.3.0, not v1.0.0. If it doesn't work, document it as Linux-Docker-only and macOS users use PyPI install path. |
| **R4** | **GHCR publishing** has surprises (auth, image size limits, multi-arch builds). | Low | Manually push a test image to GHCR during v0.5.0 or v0.6.0 to surface issues early. |
| **R5** | **PyPI Trusted Publisher** setup has friction (requires GitHub Actions OIDC). | Low | Test on a throwaway PyPI project name first if needed. |
| **R6** | **`client_secrets.json` requirement is a hard onboarding wall** that pushes away the audience the tool exists for. | Medium | Documented as a v1.0 limitation. v1.2.0 public OAuth client work is the resolution path. README explicitly cites v1.2.0 as where this gets fixed. |
| **R7** | **Appium prerequisites scare off PyPI users**, and Docker becomes the only practical path. | Low | This is fine — Docker is the primary path anyway. PyPI is secondary by design. |
| **R8** | **External contributors arrive with PRs** before the project is ready to triage them. | Low | CONTRIBUTING.md sets expectations clearly. Issue templates require enough info to triage. Worst case: PRs sit. |
| **R9** | **The tool's purpose is borderline against WhatsApp's ToS**, and the project receives a takedown request. | Low | Disclaimer is honest about the grey area. Tool is for personal data export, which is a legitimate use case in most jurisdictions. Not legal advice. |

## Cross-Cutting Concerns

These show up in multiple per-version plans and need consistent handling:

- **Activity log message conventions**: every user-facing message should be one of: status update, success, warning, error. No raw exceptions. v0.6.0 audits the existing messages; v0.3.0 onward writes new messages to the convention from day one.
- **Worker thread error handling**: every threaded worker should catch exceptions, log them via `call_from_thread` to the activity log, and post a clear "X failed: <reason>" message. The recent v0.2.0 export-pane fix is the model.
- **Persistent settings**: the Settings panel writes to the app instance, but the app doesn't currently persist settings across runs. This may need to be addressed during v0.3.0 (so users don't have to re-enter their `client_secrets.json` path on every launch). Decision deferred to v0.3.0 plan.
- **Docker image size**: every dependency added during v0.3.0 → v0.8.0 affects the image size. Be conscious about this — especially with Whisper if it pulls in heavy ML deps. (Already handled — Whisper API does not require local ML deps. Worth re-verifying when bumping deps.)

## Open Questions

These do not block the roadmap but should be answered before the relevant per-version plan is written:

1. **Should the Settings panel persist across runs?** Yes likely, but where? `~/.whatsapp_export/settings.json`? Per-user config dir? Decision: defer to v0.3.0 plan.
2. **What's the announcement plan for v1.0.0?** Defer to v0.7.0 / v1.0.0.
3. **Multi-arch Docker builds?** `linux/amd64` is mandatory; `linux/arm64` is nice-to-have for Apple Silicon hosts running Docker Desktop. Defer to v0.8.0 plan.
4. **Test coverage gate in CI?** The existing test suite has 622 unit tests and 28 integration tests. Setting a coverage gate is easy but enforcing it on PRs requires consensus. Defer to v0.8.0 plan.
5. **PR-only vs trunk-based development for the v0.x cycle itself?** Currently committing direct to main. PRs would be more disciplined but slow the cycle. Defer to ad-hoc per-version basis.

## Out-of-Roadmap Items (Future)

Captured here so they don't get forgotten but explicitly **not** part of v1.0:

- **v1.1.0 candidates**: Docker multi-arch (if not in v1.0), PyPI publishing (if not in v1.0), test coverage gate, in-TUI first-run banner with privacy/ToS notices, settings persistence improvements.
- **v1.2.0**: Public OAuth client + reduced Drive scope (`drive.file` instead of full `drive`).
- **v1.x**: Drive folder picker, Dropbox/OneDrive/S3 destinations, rclone integration as an alternative auth path, WhatsApp version compatibility table maintained over time, telemetry (opt-in only), auto-update mechanism.
- **v2.0**: anything that breaks the v1.0 CLI flag contract or settings file format.

## How to Use This Document

- **When writing a per-version plan**: cite this roadmap. If the per-version plan needs to deviate from this roadmap, propose the deviation and update this roadmap before writing the plan.
- **When making a decision during implementation**: check if it's already in the "locked" tables above. If yes, don't re-litigate. If no, decide and either record it in the per-version plan or update this roadmap.
- **When the v0.4.0 phone-local checkpoint arrives**: update the v1.0.0 scope based on the (a)/(b)/(c) decision and propagate to v0.5.0 plan.
- **When reaching v1.0.0**: archive this document and write a `v2.0.0-public-release-roadmap.md` or similar for the next major.
